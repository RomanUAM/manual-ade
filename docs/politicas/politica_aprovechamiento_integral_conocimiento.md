# Politica de Aprovechamiento Integral del Conocimiento

## Principio fundamental

El activo mas importante del proyecto no es el modelo de lenguaje.

El activo mas importante es el conocimiento acumulado durante anos de docencia, investigacion y desarrollo de materiales academicos.

Por tanto, cada archivo proporcionado por el usuario constituye patrimonio intelectual del proyecto y debe analizarse antes de generar contenido nuevo. Ningun documento puede ignorarse solo porque pertenece a otro curso, fue creado para otra actividad o parece secundario por su nombre.

## Obligacion de explorar todo el material

Antes de redactar un capitulo, practica, presentacion, pagina, evaluacion o infografia, el sistema debe revisar la base local de materiales.

Carpetas y dominios obligatorios:

- Estadistica.
- Diseno de experimentos.
- Analisis de experimentos.
- Probabilidad.
- Metodos numericos.
- Optimizacion.
- Ingenieria economica.
- Inteligencia artificial.
- Visualizacion.
- Material docente.
- Practicas.
- Presentaciones.
- Examenes.
- Simuladores.
- Fotografias.
- Articulos.
- Codigo fuente.

Ninguna carpeta puede ignorarse sin una justificacion explicita registrada en la base de conocimiento.

## Extraccion de objetos de aprendizaje

El sistema no copia documentos. Extrae conocimiento.

Cada archivo debe descomponerse en objetos de aprendizaje independientes:

- conceptos;
- ejemplos;
- ejercicios;
- figuras;
- fotografias;
- algoritmos;
- practicas;
- errores comunes;
- analogias;
- historias;
- preguntas frecuentes;
- aplicaciones industriales;
- evaluaciones;
- narrativas;
- contraejemplos;
- simuladores.

No se almacenan diapositivas como unidades pedagogicas. Se almacenan ideas reutilizables.

## Bancos reutilizables

El HAG debe mantener bancos internos para reutilizar conocimiento antes de generar contenido nuevo:

- banco de objetos de aprendizaje;
- banco de ejemplos;
- banco de figuras;
- banco de codigo;
- banco de narrativas;
- banco de errores comunes;
- banco de evaluaciones;
- banco de relaciones interdisciplinarias.

Un mismo objeto puede usarse en libro, manual, presentacion, pagina, infografia, guia del profesor y evaluacion. No debe duplicarse: debe referenciarse.

## Reutilizacion inteligente

El sistema debe buscar oportunidades de reutilizacion transversal:

- una practica de probabilidad puede explicar aleatorizacion en diseno experimental;
- un ejercicio de metodos numericos puede apoyar optimizacion de parametros;
- una simulacion de inteligencia artificial puede ilustrar disenos factoriales;
- una fotografia de laboratorio puede enriquecer libro, presentacion, pagina e infografia;
- un error real de estudiante puede convertirse en actividad diagnostica, caso comparativo y pregunta de evaluacion.

La generacion de contenido nuevo es el ultimo recurso. Primero debe agotarse el conocimiento existente.

## Integracion entre asignaturas

Cuando un concepto aparezca en varias asignaturas, el sistema debe construir una narrativa interdisciplinaria.

Ejemplo:

Probabilidad -> Inferencia estadistica -> Diseno de experimentos -> ANOVA -> Control estadistico -> Aprendizaje automatico -> Ciencia de datos.

El estudiante debe percibir que las asignaturas forman parte de una misma historia intelectual y profesional.

## Presentaciones

Cada presentacion debe analizarse por completo. No basta copiar diapositivas.

El sistema debe identificar:

- figuras utiles;
- esquemas;
- animaciones;
- ejemplos;
- explicaciones verbales implicitas;
- preguntas de clase;
- errores mostrados en clase;
- secuencias visuales que puedan convertirse en narrativa de libro.

Despues debe expandir ese contenido para integrarlo al libro, manual, pagina, evaluacion o guia.

## Practicas

Cada practica debe responder:

- que conceptos ensena;
- que errores permite observar;
- que fotografias pueden incorporarse;
- que resultados pueden reutilizarse;
- que tablas sirven como ejemplos;
- que ejercicios pueden derivarse;
- en que otros capitulos puede aparecer.

Ninguna practica debe permanecer aislada.

## Codigo

El codigo no pertenece solo a un archivo fuente. Pertenece al conocimiento.

Cada algoritmo debe registrarse con:

- nombre;
- proposito;
- lenguaje;
- tema relacionado;
- nivel de dificultad;
- explicacion;
- pseudocodigo;
- ejemplo de uso;
- ejercicios asociados;
- posible simulador;
- documentacion.

Si falta codigo para un tema, se marca como pendiente. Si existe codigo y no se explica, el Director HAG debe rechazar la entrega.

## Errores comunes y comparaciones

Los errores son recursos didacticos principales.

Cada tema debe buscar y registrar:

- que hace normalmente un estudiante;
- por que parece correcto;
- que problema genera;
- como corregirlo;
- ejemplo incorrecto;
- ejemplo correcto;
- evidencia visual cuando exista.

Siempre que existan ejemplos buenos y malos, deben presentarse juntos.

## Criterio de rechazo del Director HAG

El Director HAG debe rechazar cualquier capitulo o entrega cuando detecte que:

- existen documentos relacionados no consultados;
- existen fotografias no utilizadas;
- existen practicas relacionadas no integradas;
- existen presentaciones con mejores ejemplos;
- existen algoritmos no incorporados;
- existe codigo no explicado;
- existen ejercicios reutilizables ignorados;
- no existe banco de objetos de aprendizaje actualizado;
- no existe trazabilidad entre fuente, objeto, uso y recurso final.

## Objetivo final

El sistema no debe escribir mas.

Debe pensar mejor.

Su mision es transformar anos de experiencia docente e investigadora en un ecosistema academico integrado donde ningun documento, fotografia, practica, presentacion, algoritmo o experiencia de laboratorio permanezca aislado o desaprovechado.

Cada nuevo recurso debe aumentar el valor de toda la coleccion y no solo del documento que se esta desarrollando.
